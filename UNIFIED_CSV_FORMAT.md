# LvKong_1C_V2.5_A484_A485_100_00B_unified.csv 格式说明

## 概述

这是一个统一的 CSV 表，将 CAN 数据库的 **消息（Message）** 和 **信号（Signal）** 信息整合到一个表中。每一行代表一个信号，并包含其所属消息的完整信息。

**总列数**：19 列  
**数据行数**：82 行（共 82 个信号）

---

## 列定义

### 第一部分：消息信息（6 列）

这些列为该信号所属消息的信息。**同一消息的多个信号，消息信息相同**。

| 列名 | 数据类型 | 必填 | 说明 | 示例 |
|------|---------|------|------|------|
| **消息ID** | 16进制/10进制 | ✓ | CAN 消息 ID，十六进制或十进制格式 | `0x18FF5678` 或 `419387000` |
| **消息名称** | 字符串 | ✓ | CAN 消息的名称 | `MCU2_STATUS5` |
| **消息长度** | 整数 | ✓ | 消息数据长度（字节数），范围 0-8（标准 CAN）或更大（CAN FD） | `8` |
| **周期时间(ms)** | 整数 | ✗ | 消息的发送周期（毫秒），可为空 | `10` 或空 |
| **发送者** | 字符串 | ✗ | 消息的发送节点名称，可为空则默认为 `Vector__XXX` | `MCU2` 或 `HCU` |
| **消息备注(JSON)** | JSON 字符串 | ✗ | 消息的注释，JSON 格式，可为空 | `{"description":"Motor status"}` |

### 第二部分：信号信息（13 列）

每一行的信号定义。同一消息的不同信号，这部分信息不同。

| 列名 | 数据类型 | 必填 | 说明 | 示例 |
|------|---------|------|------|------|
| **信号名称** | 字符串 | ✓ | CAN 信号的名称，全局唯一性建议 | `MCU1_VoltD` |
| **起始位** | 整数 | ✓ | 信号在 CAN 消息中的起始比特位 (0-63) | `0` |
| **长度(bit)** | 整数 | ✓ | 信号长度（比特数），范围 1-64 | `16` |
| **字节序** | `little_endian` 或 `big_endian` | ✓ | 字节序（小端 = Intel，大端 = Motorola） | `little_endian` |
| **有符号** | `是` 或 `否` | ✓ | 是否为有符号数（补码表示）| `否` |
| **初值** | 数字 | ✗ | 信号的初始值，可为空 | `0` 或空 |
| **缩放系数** | 浮点数 | ✓ | 信号的缩放系数，计算：`物理值 = 原始值 × scale + offset` | `1` 或 `0.1` |
| **偏移** | 浮点数 | ✓ | 信号的偏移值 | `-30000` 或 `0` |
| **最小值** | 浮点数 | ✗ | 信号物理值的最小范围，可为空 | `0` 或空 |
| **最大值** | 浮点数 | ✗ | 信号物理值的最大范围，可为空 | `65535` 或空 |
| **单位** | 字符串 | ✗ | 信号的物理单位，可为空 | `V`、`rpm`、`Nm`、`degree Celsius` |
| **接收者** | 字符串 | ✗ | 信号接收节点，多个用逗号分隔，可为空 | `HCU` 或 `MCU1, MCU2` |
| **备注(JSON)** | JSON 字符串 | ✗ | 信号的注释，JSON 格式，可为空 | `{"0":"Ready","1":"Enable"}` |

## 数据格式规范

### 1. 消息 ID（消息ID 列）

支持两种格式：
- **十六进制**：以 `0x` 开头，如 `0x18FF5678`
- **十进制**：直接数字，如 `419387000`

扩展帧判定：ID > 0x7FF 为扩展帧

**示例**：
```
0x18FF5678  (十六进制扩展帧)
0x4A88      (十六进制标准帧)
419387000   (十进制)
```

### 2. 数值字段

- **整数字段**（消息长度、起始位、长度）：整数，不能有小数点
- **浮点字段**（缩放系数、偏移、最小值、最大值）：支持整数和浮点数

**示例**：
```
缩放系数: 1, 0.1, -30000
偏移: 0, -3200, 2
```

### 3. 字节序字段

**值**：
- `little_endian`（小端，Intel 格式）
- `big_endian`（大端，Motorola 格式）

### 4. 有符号字段

**值**：
- `是` ：有符号数（二补码表示）
- `否` ：无符号数

### 5. 接收者字段

多个接收者用逗号分隔，会自动 trim() 空格：
```
MCU1
MCU1, MCU2
HCU, MCU1, MCU2
```

### 6. 注释字段（JSON 格式）

两个注释字段都支持 JSON 格式，脚本会自动解析并转换：

#### 消息备注(JSON)

任意 JSON 对象，如：
```json
{}
{"description":"Motor status"}
{"period":"10ms"}
```

#### 信号备注(JSON) - 三种格式

**格式1：单行描述**
```json
{"description":"电机转速正负表示电机转向"}
```
输出为原文本：
```
电机转速正负表示电机转向
```

**格式2：纯数字键（十进制）**
```json
{"0":"Ready","1":"Enable","2":"PowerUp","3":"Error"}
```
输出格式为 `key = value` 每行一个：
```
0 = Ready
1 = Enable
2 = PowerUp
3 = Error
```

**格式3：二进制字符串键**
```json
{"00":"空档位","01":"前进档位","10":"后退档位"}
```
输出格式为 `key:value` 每行一个（注意使用冒号）：
```
00:空档位
01:前进档位
10:后退档位
```

#### 自动转换规则

脚本会自动识别和转换注释格式：
- 如果 key 是纯数字（如 "0", "1", "2"），使用 `key = value` 格式
- 如果 key 是二进制字符串（如 "00", "01", "10"），使用 `key:value` 格式
- 如果只有 `"description"` 字段，直接使用其值作为注释
- 如果不是有效 JSON，保持原文本不变

0x18FF5678,MCU2_STATUS5,8,10,MCU2,{"description":"Motor status"},MCU1_VoltD,0,16,little_endian,否,,1,-30000,,,V,HCU,{"unit":"Volt"}

### 空字段处理

脚本采用以下策略处理空字段：
- **可选字段为空**：保持为空（`""`）
- **最小值/最大值为空**：被解析为 `None`（无范围限制）
- **发送者为空**：默认设为 `Vector__XXX`
- **接收者为空**：被解析为空列表 `[]`
- **注释为空**：被保留为空字符串

## 编辑建议

### ✓ 可以安全修改的字段

- **周期时间**：修改后会自动更新消息注释中的 `period XXms` 信息
- **发送者、接收者**：节点名称
- **单位**：信号物理单位
- **缩放系数、偏移**：数值转换参数
- **初值、最小值、最大值**：范围限制参数
- **消息备注、信号备注**：JSON 格式的注释信息

### ✗ 不建议修改的字段

- **消息 ID**：改变后会产生新消息
- **消息名称**：对应 DBC 中的消息定义
- **消息长度**：必须与 DBC 定义一致
- **信号名称**：对应 DBC 中的信号定义
- **起始位、长度、字节序、有符号**：改变会导致信号错误

## csv2dbc.py 处理流程

1. **读取 CSV**：按行读取，以消息 ID 为 key 存储数据
2. **消息去重**：相同消息 ID 的行只创建一次消息对象，信号则全部加入
3. **数据类型转换**：
   - 消息 ID：十六进制转整数
   - 整数字段：转 int
   - 浮点字段：转 float（缺失则为 None）
   - 接收者：按逗号分割并 strip
4. **周期信息处理**：从"周期时间(ms)"列提取周期信息，格式化为 `period XXms` 添加到消息注释中
5. **注释转换**：
   - **消息注释**：尝试将注释解析为 JSON，如果是有效 JSON 则保留，否则保持原文
   - **信号注释**：尝试将注释解析为 JSON，并根据 key 类型自动转换格式：
     - 纯数字键（0, 1, 2...）→ `0 = value\n1 = value...` 格式
     - 二进制字符串键（00, 01, 10...）→ `00:value\n01:value\n10:value` 格式
     - 仅含 "description" 字段 → 直接使用文本值
     - 非 JSON 格式 → 保持原文本不变
6. **Conversion 对象创建**：使用 `LinearConversion` 对象处理 scale 和 offset
7. **字节序判定**：`little_endian` 保留，其他视为 `big_endian`
8. **生成 DBC**：使用 cantools 库的 `as_dbc_string()` 方法
9. **验证**：加载生成的 DBC 文件并验证消息数量一致性

## 常见错误和排查

| 错误 | 原因 | 解决方案 |
|------|------|--------|
| 消息数量不匹配 | 消息 ID 或名称被修改 | 检查第 1、2 列，保持与原始 DBC 一致 |
| 信号缺失 | 某行被删除或消息 ID 错误 | 检查所有 82 行，验证消息 ID |
| 数据类型错误 | 数值字段包含非数字字符 | 检查缩放系数、偏移等数字字段 |
| JSON 解析失败 | 注释字段格式不正确 | 验证 JSON 格式，或留空 |
| 无法生成 DBC | 依赖文件缺失（原始 DBC）| 确保 `LvKong_1C_V2.5_A484_A485_100_00B.dbc` 存在 |
